2 IBM MAMMAL — The Teacher Model

ibm-research/biomed.omics.bl.sm.ma-ted-458m

MAMMAL (Molecular Aligned Multi-Modal Architecture and Language) is IBM Research's unified biomedical foundation model published in npj Drug Discovery (May 4, 2026). It was pre-trained on 2 billion biological samples across proteins, small molecules, and single-cell transcriptomics — making it the broadest open-source biomedical foundation model available.

What It Handles
  • DTI Prediction — protein FASTA + SMILES → binding affinity (pKd)
  • Toxicity — ClinTox, carcinogenicity from SMILES alone
  • Cancer-Drug Response — SMILES + scRNA-seq → IC₅₀
  • SMILES generation — infilling and design tasks
Key insight: IBM MAMMAL cannot be merged with Gemma — one is encoder-decoder (458M), the other is decoder-only (2B). But MAMMAL's binding affinity predictions are SOTA. We use it as a data quality filter: generate a large pool of synthetic molecule-target pairs, run them through MAMMAL, and only keep the ones that score ≥ 0.70 binding probability. Gemma v2 then trains only on high-affinity, MAMMAL-validated gold-standard pairs.

Why this works

Raw ChEMBL contains low-affinity noise (Kd > 10µM compounds that barely bind). Training on this teaches Gemma to generate chemically valid but target-irrelevant molecules. MAMMAL's DTI model removes all low-quality pairs before they enter the training set — Gemma v2 only sees protein-molecule pairs where binding is actually predicted to be strong.

What MAMMAL filters on

  • Binding probability ≥ 0.70 (strict affinity gate)
  • ClinTox pass (no predicted clinical toxicity)
  • BBBP if CNS target (blood-brain barrier penetration)
  • Carcinogenicity screen (additional safety filter)

MAMMAL Filter Script

from mammal.model import Mammal
from fuse.data.tokenizers.modular_tokenizer.op import ModularTokenizerOp
import torch

# Load once — keep in memory during the full filter pass
model_id = "ibm-research/biomed.omics.bl.sm.ma-ted-458m"
mammal   = Mammal.from_pretrained(model_id).cuda().eval()
tok_op   = ModularTokenizerOp.from_pretrained(model_id)

def passes_mammal_filter(fasta_seq: str, smiles: str, threshold: float = 0.70) -> bool:
    """Return True if IBM MAMMAL predicts binding probability >= threshold."""
    sample = {"encoder_input": (
        f"<@TOKENIZER-TYPE=AA><BINDING_AFFINITY_CLASS><SENTINEL_ID_0>"
        f"<MOLECULAR_ENTITY><MOLECULAR_ENTITY_GENERAL_PROTEIN>"
        f"<SEQUENCE_NATURAL_START>{fasta_seq}<SEQUENCE_NATURAL_END>"
        f"<MOLECULAR_ENTITY><MOLECULAR_ENTITY_SMALL_MOLECULE>"
        f"<SMILES_START>{smiles}<SMILES_END><EOS>"
    )}
    tok_op(sample_dict=sample, key_in="encoder_input",
           key_out_tokens_ids="enc_tokens",
           key_out_attention_mask="enc_mask")

    enc_t = torch.tensor(sample["enc_tokens"]).unsqueeze(0).cuda()
    enc_m = torch.tensor(sample["enc_mask"]).unsqueeze(0).cuda()

    with torch.no_grad():
        out = mammal.generate([sample], output_scores=True,
                              return_dict_in_generate=True, max_new_tokens=5)

    pred = tok_op._tokenizer.decode(out["cls_pred"][0])
    try:
        return float(pred) >= threshold
    except ValueError:
        return False

# Run over ChEMBL pairs — keep only gold-standard entries
gold_pairs = [p for p in raw_chembl_pairs
              if passes_mammal_filter(p["fasta"], p["smiles"])]

Why flipping to 65% target-conditioned fixes the core problem: The model can no longer minimize its loss function by memorizing generic high-QED drug strings from MOSES. Every training example now connects a specific protein sequence to a specific molecule. The model is forced to learn the protein-to-scaffold mapping rather than just chemical syntax.

5 New Training Format — Multi-Modal ChatML

v1 used a flat prompt format. v2 wraps every example in structured ChatML with an explicit rationale chain — forcing the model to reason about the target before generating the SMILES string.

v1 Format (flat)

Generate novel drug-like SMILES similar
to: CC1CC1NC(=O)C(F)(F)F
Target: SGVTLFVALYDYE...
Diversity: 0.5
Rules: ...
Generate exactly 8 SMILES:

v2 Format (ChatML + rationale)

<|im_start|>user
Task: Design a small molecule drug candidate
that binds to the following protein target.
Target FASTA: MERALAPVQR...
Desired: High selectivity, QED > 0.55,
low cardiotoxicity.
<|im_end|>
<|im_start|>assistant
Rationale: The binding pocket contains a
hydrophobic cleft with polar residues.
An aromatic core maximises pi-pi stacking.
SMILES: CC1=C(NC2=NC=CC=N2)C=CC=C1
<|im_end|>
Why rationale-first matters: Chain-of-thought before generation forces the model to encode target context before decoding the SMILES. This directly improves target-conditioned generation — the model can't shortcut to a memorized scaffold because it has to "explain" the binding first.

6 QLoRA v2 Configuration

Parameterv1 Valuev2 ValueReason
LoRA Rank (r) 16 32 Higher rank = more capacity to model complex protein→molecule mappings. Doubles trainable parameters without changing base model.
Alpha (α) 32 64 α/r = 2 ratio maintained. Controls effective learning rate of adapter — scales proportionally with rank.
Target modules q,k,v,o,gate,up,down q,k,v,o,gate,up,down Same 7 modules — proven coverage. No change needed.
Trainable params ~20M (~1%) ~40M (~2%) Doubling from rank 16→32. Still well within 4-bit VRAM budget on GB10.
Learning rate 2e-4 1.5e-4 Lower LR for higher rank — prevents overfitting on larger adapter. Cosine decay retained.
Batch size 4 × 8 accum = 32 8 × 4 accum = 32 Same effective batch. More samples per step improves gradient quality for the target-conditioned data.
Weight decay 0 0.01 Added regularization — prevents the adapter from overfitting the 65K target pairs.
Epochs 3 3 Unchanged. Cosine decay over 3 epochs is well-calibrated for this dataset size.
use_rslora False True RS-LoRA normalises rank by √r — more stable training at r=32 vs naive LoRA scaling.

v2 QLoRA Code

from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name   = "google/gemma-4-e2b-it",
    max_seq_length = 4096,
    dtype          = None,
    load_in_4bit   = True,
)

model = FastLanguageModel.get_peft_model(
    model,
    r              = 32,          # upgraded from 16
    lora_alpha     = 64,          # upgraded from 32
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                      "gate_proj", "up_proj", "down_proj"],
    lora_dropout   = 0,
    bias           = "none",
    use_gradient_checkpointing = "unsloth",
    use_rslora     = True,        # RS-LoRA for stable high-rank training
    random_state   = 3407,
)

8 Execution Plan — Step by Step

Step 1 · Data Pipeline

Build the MAMMAL-filtered dataset

Download ChEMBL v33 + BindingDB. Filter to Kd/Ki < 100nM. Run IBM MAMMAL filter (binding prob ≥ 0.70). Serialize to ChatML with rationale generation. Target: 80–100K gold pairs. Estimated time: 1–2 days.

Step 2 · SMolInstruct Fix

Fetch via direct Parquet download

Bypass the deprecated loading script using huggingface_hub.hf_hub_download() to pull raw Parquet files. Sample 50K chemistry instruction pairs from 3M+ available. Addresses the v1 fallback to Mol-Instructions.

Step 3 · Scale Deep2Lead Custom

240 → 5,000 target-specific samples

For all 20 production targets, generate diverse instruction formats: rationale+SMILES, property-conditioned, scaffold-hopping, bioisostere substitution. Run MAMMAL filter on all generated examples. Remove weak binders.

Step 4 · Train v2 on GB10

QLoRA r=32, α=64, RS-LoRA

Same DGX Spark GB10 hardware. With r=32, expect ~22.8 GB VRAM (same — rank doubles adapter size but base stays 4-bit quantized). Training time: estimated 6–8 hrs at 100K samples. Monitor loss on target-conditioned subset separately.

Step 5 · MAMMAL Eval Loop

Post-training quality validation

Generate 200 molecules across 10 targets. Run MAMMAL DTI on each. Report: % scoring ≥ 0.65, validity, uniqueness, QED, SAS. Compare directly to v1 on the same 5 benchmark targets from the finetune report.

Step 6 · vLLM Serving

Replace FastAPI with vLLM on dgx1

Switch serve_finetuned.py to vLLM backend with continuous batching. Expected latency: ~4–5s vs current ~12s. Makes fine-tuned model competitive with production on speed — removes the last remaining weakness.
