1. Project Overview

Deep2Lead is a drug discovery pipeline developed at Georgia Tech. The research (arXiv:2108.05183) demonstrates that a Variational Autoencoder (VAE) trained on the ZINC drug database can generate novel drug-like molecules from a seed SMILES string by perturbing the latent space with Gaussian noise. Candidate molecules are then scored for Drug-Target Interaction (DTI) using Stanford's DeepPurpose library.

V2 replaces all external dependencies with fully self-owned components running on in-house infrastructure (DGX Spark), making the pipeline suitable for confidential research, offline use, and student teaching without relying on third-party APIs or licenses.

2. V1 System Analysis

2.1 What V1 Does

The V1 system exposes three REST endpoints:

EndpointMethodInputOutput
/api/smilePOSTSMILES string + noise floatSingle perturbed SMILES + QED, SAS, LogP scores
/api/random/smilesGETNoneSet of up to 6 random valid SMILES from ZINC distribution
/api/random/smiles/v2POSTSMILES + noise + attempts + seedSet of N perturbed SMILES molecules

2.2 V1 Core Algorithm

  1. Canonicalize input SMILES using RDKit
  2. One-hot encode SMILES → character matrix
  3. Pass through pre-trained Keras encoder → 196-dim latent vector z
  4. Add Gaussian noise to z (noise magnitude controlled by user)
  5. Pass noisy z through Keras decoder → one-hot matrix
  6. Convert one-hot back to SMILES string
  7. Validate with RDKit; retry if invalid
  8. Compute QED (drug-likeness), SAS (synthetic accessibility), LogP (lipophilicity)

2.3 V1 Problems

  • Python 3.6 + TensorFlow 1.x — both end-of-life, security vulnerabilities, incompatible with modern libraries
  • Keras .h5 model files — no versioning, format deprecated in TF2
  • DeepPurpose external dependency — requires internet access, version-locked, no customization
  • Hardcoded paths — /Users/user/Documents/BD4H/... in production code
  • No amino acid sequence input — DTI is implicit via DeepPurpose only
  • Monolithic app.py — all logic in one 460-line file
  • No UI — redirects to Swagger petstore (external URL)
  • No protein target input in API — the drug.py file is incomplete/unused

3. V2 Goals & Principles

3.1 Primary Goals

  • Full vertical ownership — no external API calls during inference
  • Accept amino acid sequence (protein target) + seed SMILES as joint input
  • Use Gemma4-E4B on DGX for SMILES generation (replace VAE)
  • Custom DTI predictor replacing DeepPurpose
  • Modern Python stack (3.11+, PyTorch 2.x)
  • Modular codebase suitable for Georgia Tech student teaching
  • Self-contained web UI, no redirects to external tools

3.2 Design Principles

  • Service isolation — each concern (generation, validation, DTI) lives in its own module
  • Fail loudly — no silent fallbacks; errors surface to the UI with clear messages
  • Stateless API — no session state, each request is self-contained
  • Incremental deployment — Phase 1 works without fine-tuning; Phase 2 adds fine-tuned models
  • Separate static assets — all HTML, CSS, JS in separate files per project convention

4. Drug Discovery Pipeline

The V2 pipeline for a single request:

1
User Input
Amino acid sequence (protein target)
Seed SMILES string
Parameters: noise level, num candidates
↓
2
Input Validation
Validate SMILES with RDKit
Validate amino acid sequence (20 standard AA chars)
Canonicalize SMILES
↓
3
Molecule Generation (Gemma4)
Send prompt to Gemma4 API on DGX (port 9001)
Prompt includes: seed SMILES, protein context, noise guidance
Request N candidate SMILES strings in structured output format
↓
4
SMILES Filtering
Parse Gemma4 output to extract SMILES
Validate each with RDKit (discard invalid)
Deduplicate; remove seed molecule
↓
5
Property Calculation
QED — drug-likeness (0–1, higher = better)
SAS — synthetic accessibility score (1–10, lower = easier to make)
LogP — lipophilicity (Crippen method)
Tanimoto similarity to seed (Morgan fingerprint)
Molecular weight, H-bond donors/acceptors (Lipinski filter)
↓
6
DTI Prediction
Encode protein: ESM-2 (150M) → 640-dim embedding
Encode drug: RDKit ECFP4 → 2048-bit fingerprint
MLP classifier → binding affinity score (0–1)
(Phase 1: rule-based scoring; Phase 2: trained MLP)
↓
7
Ranking & Response
Score = w1×DTI + w2×QED − w3×SAS + w4×Tanimoto
Return top-K candidates sorted by composite score
Include all property values and DTI score per candidate

8.2 Prompt Engineering Strategy

Gemma4 does not natively understand molecular chemistry without fine-tuning. For Phase 1 (pre-fine-tuning), we use structured prompting to guide it toward valid SMILES generation: 

SYSTEM:
You are a computational chemistry assistant specializing in drug discovery.
Generate structurally similar but novel drug-like molecules.
Rules:
- Output ONLY valid SMILES strings, one per line, no explanations
- Generated molecules must be structurally similar to the seed (Tanimoto > 0.4)
- Obey Lipinski's Rule of Five: MW < 500, LogP < 5, HBD ≤ 5, HBA ≤ 10
- Make small structural modifications: substitute atoms, add/remove small groups
- Do not simply copy the seed molecule

USER:
Seed SMILES: {canonicalized_smile}
Target protein (first 50 AA): {amino_acid_seq[:50]}
Noise level: {noise} (0=minimal change, 1=large change)

Generate exactly {num_candidates} novel SMILES strings:

8.3 Response Parsing

Gemma4 output will be free text. The molecule_generator.py service extracts SMILES using:

  1. Regex to find lines matching common SMILES patterns (atoms, bonds, brackets)
  2. RDKit MolFromSmiles() as the final validity gate
  3. If fewer than requested valid SMILES are returned, retry the API call up to 3 times

8.4 Fallback (When DGX Is Unavailable)

If the DGX health check fails, the system falls back to the V1 VAE method (if the Keras model files are present) or returns an error with a clear message. This ensures the app remains useful for students even during GPU downtime.

9. DTI Replacement Strategy

9.1 Why Replace DeepPurpose?

DeepPurpose (Stanford, 2020) is an excellent library but it requires downloading pretrained models from the internet, has pinned dependencies that conflict with modern Python, and cannot be fine-tuned without forking the repo. V2 needs a DTI component that can be retrained on new data, runs fully offline, and integrates cleanly into the Flask service layer.

9.2 Chosen Approach: ESM-2 + ECFP + MLP

ComponentRoleOutput Dimension
ESM-2 (150M, Facebook)Protein sequence encoder640-dim vector (mean pooled)
RDKit ECFP4Drug SMILES encoder (Morgan fingerprint, radius=2)2048-bit binary vector
MLP ClassifierPredict binding affinityScalar score 0–1

9.3 MLP Architecture

Input: concat(protein_vec[640], drug_vec[2048]) = 2688-dim

MLP:
  Linear(2688, 1024) → BatchNorm → ReLU → Dropout(0.3)
  Linear(1024, 256)  → BatchNorm → ReLU → Dropout(0.2)
  Linear(256, 64)    → ReLU
  Linear(64, 1)      → Sigmoid

Output: binding probability ∈ [0, 1]

9.4 Training Data

DatasetSizeUse
BindingDB (Ki)~1.8M pairsPrimary training data
Davis~30K pairsValidation / benchmark
KIBA~118K pairsAdditional training

Training runs on the DGX using the existing Unsloth Python environment. Scripts placed in scripts/train_dti.py. Expected training time: ~4–8 hours on GB10 GPU.

9.5 Phase 1 Interim: Rule-Based DTI

Before the MLP is trained, the DTI service returns a heuristic score based on: Lipinski compliance (pass/fail), QED, and Tanimoto similarity to a set of known active compounds from the ZINC database. This gives students a working pipeline on day one.

9.6 ESM-2 Availability

ESM-2 (150M parameters) is available via HuggingFace (facebook/esm2_t30_150M_UR50D). The model and tokenizer are downloaded once to the DGX server and cached locally, then served as a Python import in dti_predictor.py. No internet required after initial setup.

12. Fine-Tuning Plan (Phase 3 Detail)

12.1 Dataset Construction

Use the existing ZINC dataset (250k_rndm_zinc_drugs_clean_3.csv already in V1 model files) to build instruction-response pairs: 

Instruction: "Given the SMILES string {seed}, generate a structurally
              similar drug-like molecule with slight modifications.
              Return only the SMILES string."

Response: "{similar_smiles}"

Pairs are selected where Tanimoto(seed, response) ∈ [0.4, 0.9], ensuring the model learns to make meaningful but not trivial changes. Target: 50,000–100,000 training pairs.

14. Risks & Mitigations

RiskLikelihoodImpactMitigation
Gemma4 generates mostly invalid SMILES before fine-tuning HighMedium Retry logic (3 attempts); fallback to VAE if <2 valid SMILES returned; improve prompt iteratively
DGX unavailable / GPU OOM (multiple users) MediumHigh Health check before each request; graceful error message to UI; V1 VAE fallback mode
ESM-2 slow for long sequences LowLow Cache protein embeddings by sequence hash; truncate to first 512 AA (standard practice)
DTI model generalization (new protein families) MediumMedium Show confidence interval in UI; label as "research estimate, not clinical prediction"
Fine-tune dataset quality (ZINC SMILES pairs too similar) MediumMedium Filter: require Tanimoto ∈ [0.4, 0.9]; include scaffold-hopped pairs from ChEMBL
Students misuse app for clinical claims LowHigh Prominent disclaimer in UI: "Research tool only — not for clinical decision-making"
