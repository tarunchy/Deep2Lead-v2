PathoHunt 3D investigation

Attack API fix and game UX improvement plan

Scope: analyze the 400 BAD REQUEST seen while firing molecules at /game/pathohunt-3d/influenza_na, identify the likely root cause, and propose implementation changes for review before touching production code.

Root cause Session reaches won, then later attack calls reuse the same completed session.
Secondary bug Multiple attack requests can overlap, creating duplicate attack numbers and stale state.
UX impact The player sees "API ERROR" instead of victory, completed-session, or retry guidance.

Observed Evidence

Server log pattern: session 633090a3-7a31-4dd0-96e9-46d3b2280875 accepted several POST /attack requests with HTTP 200, then subsequent POST /attack requests returned HTTP 400 with a 34 byte body. A 34 byte JSON body matches {"error":"Session is not active"}.
Database state: that session exists with status = won, best_score = 0.704, and win_threshold = 0.67. The first later 400 is therefore expected from the current backend guard, but it is handled badly by the frontend. 
{
  found: true,
  status: "won",
  attacks_count: 3,
  best_score: 0.704,
  boss_current_hp: 266.78,
  win_threshold: 0.67
}
Concurrency evidence: the same session has five stored attack rows but attacks_count is only 3, and attack numbers are duplicated. That indicates overlapping requests across gunicorn workers are reading and committing stale session state. 
[
  [1, 0.5680, 18.89, 281.11, true],
  [1, 0.5744, 19.06, 280.94, true],
  [2, 0.5744,  7.44, 273.67, true],
  [3, 0.7040, 18.16, 255.51, true],
  [3, 0.5680,  6.89, 266.78, false]
]

Why It Happens

  1. The route maps any game-service ValueError to HTTP 400. See v2/api/game.py:99-109.
  2. The service rejects completed sessions. v2/services/game_service.py:134-135 raises Session is not active when the session is already won or lost.
  3. The backend declares victory by score threshold, not HP depletion. v2/services/game_service.py:240-248 marks the session as won when a molecule beats the known drug by 5 percent, even if boss HP is still high. That is valid game logic, but the UI still presents boss HP as the main win metaphor.
  4. The frontend starts the expensive backend attack immediately at launch. v2/static/js/game_battle_3d.js:1005-1015 calls /attack as soon as the projectile is fired, before the projectile hits, misses, or gets blocked.
  5. Blocked shots unlock the player while the original backend request can still be running. v2/static/js/game_battle_3d.js:1921-1931 sets attackLocked = false when a spore blocks a projectile. The user can fire again while a previous /attack is still mutating the same session.
  6. The frontend hides the actual error. v2/static/js/game_battle_3d.js:1052-1058 treats all errors as "API ERROR - minimal damage applied" and does not show data.error or recover to victory/loss state.

Fix Plan

P0 backend

Return meaningful completed-session responses

  • Change inactive-session handling from generic HTTP 400 to either HTTP 409 with a structured code, or HTTP 200 with current session state when status is won or lost.
  • Recommended payload: { code, error, session, won, lost }. For won, include best attack data so the client can show victory instead of "API ERROR".
  • Keep invalid SMILES and missing target as 400. Treat completed session as state conflict, not bad input.
P0 frontend

Stop masking attack errors

  • In launchAttack, parse r.ok and persist status, code, and error on proj.apiResult.
  • In applyAttackResult, if the response says SESSION_NOT_ACTIVE and the returned session is won, call onVictory. If it is lost, show the loss screen.
  • For real network failures, show a clear retry toast and do not apply fake minimal damage. Fake damage hides data integrity bugs.
P0 state

Remove overlapping session mutations

  • Use a row lock in execute_attack: query the session with with_for_update() inside one transaction before reading status, best_score, boss_current_hp, and attacks_count.
  • Add a uniqueness rule or sequence-safe counter for GameAttack.attack_number per session.
  • Consider a client_attack_id UUID in the request and a unique DB constraint so retries are idempotent instead of double-scoring.
P0 game logic

Only score actual boss hits

  • Move the /attack API call from projectile launch to the moment the projectile parks on/hits the boss.
  • If a projectile misses, hits a green cell, or is blocked by a spore, do not call /attack and do not mutate the server session.
  • This aligns player-visible action with persisted game state and eliminates many race windows.
Recommended implementation order: first structured error handling, then call-on-hit, then backend row locking/idempotency. This gives immediate user-visible relief while still closing the deeper race condition.

UI/UX Improvement Plan

Issue Current behavior Recommended fix
Win condition mismatch The UI emphasizes boss HP, but the backend wins immediately when score beats known drug threshold. In the observed session, victory happened with boss HP still around 267. Rename the main objective to "Discovery Threshold" or make HP depletion the actual win condition. My recommendation: keep threshold victory, but make it first-class in the HUD and victory transition.
Too much permanent HUD chrome Top bars, baseline strip, right sidebar, combat log, instructions, science card, docking guide, and deck compete for attention. Reduce the active HUD to: target status, selected molecule, discovery progress, fire state. Move environment, difficulty, missile, and trajectory controls into a settings drawer.
Error feedback is not actionable All API failures become "API ERROR - minimal damage applied". Use specific states: "Mission already completed", "Molecule generation timed out", "Invalid SMILES", "Network retry available". Include a retry button where safe.
Waiting state feels disconnected The docking guide appears after impact, but the backend request was already started at launch. Start the backend request at impact and let the docking guide become the honest loading state. Show phases as real progress: queued, generating, scoring, result.
Mobile/responsive support is weak game_battle_3d.css has fixed desktop dimensions and no meaningful media queries for the game HUD. Add breakpoints for tablet/mobile, collapse side controls, enlarge touch targets, and make deck cards horizontally scrollable below 768px.
Visual hierarchy Everything is cyan-on-dark, which fits sci-fi but makes controls and data feel equally important. Keep the sci-fi style but assign semantic colors: teal for actions, amber for science/threshold, red for danger, green for success. Increase contrast on secondary text currently near #333/#445.

Concrete Change List For Next Pass

Backend files

  • v2/api/game.py: add structured attack error responses and use 409 for completed-session conflict if not returning final state as 200.
  • v2/services/game_service.py: add locked session read/update, idempotency key support, consistent attack number assignment, and a helper for final session result.
  • v2/models/game_models.py: add optional client_attack_id and unique constraints if we choose idempotency.
  • v2/migrations/versions/...: create migration for any new column/constraint.

Frontend files

  • v2/static/js/game_battle_3d.js: move API call to hit resolution, add AbortController or request registry, handle non-OK responses, and route completed sessions to victory/loss screens.
  • v2/templates/game_battle_3d.html: simplify always-visible HUD, add a clear discovery objective panel, and move advanced controls into a drawer.
  • v2/static/css/game_battle_3d.css: add responsive breakpoints, stronger text contrast, clearer control hierarchy, and less persistent visual noise.

Validation Plan

  • Backend unit test: attacking a completed session returns a structured completed-session response, not a generic 400.
  • Backend concurrency test: two simultaneous attacks on one active session cannot create duplicate attack numbers or stale boss_current_hp.
  • Frontend test: mock a completed-session response and verify the client opens victory/loss UI instead of logging "API ERROR".
  • Gameplay test: blocked and missed projectiles do not create GameAttack rows.
  • Manual smoke test: run v2/run.sh restart, play /game/pathohunt-3d/influenza_na, fire until win, then confirm no repeated console 400s.

Risk Notes

  • Moving API scoring from launch to impact makes the player wait after impact, but it is more truthful and prevents blocked shots from scoring in the database.
  • Row locks protect session integrity but can make concurrent attacks wait. That is acceptable because one session should resolve one scoring event at a time.
  • Changing win-condition language matters. If the game says "boss HP", players expect HP zero to win. If the actual win is discovery threshold, the UI should say so directly.
